# Phase 5: 打包分发

> **计划来源**: docs/phase_5_electron/plan_1.md
> **阶段目标**: 配置 electron-builder，生成 Windows 和 macOS 安装包
> **前置依赖**: Phase 1-4 全部完成

---

## 项目背景

### 概述

将 Electron 应用打包为可分发的安装包，支持 Windows 和 macOS 平台。后端服务随 Electron 应用自动启动（仅生产环境）。

### 核心需求

| 需求 | 描述 | 优先级 |
|------|------|--------|
| 后端自动启动 | 生产环境 Electron 应用启动时自动启动后端服务 | P1 |
| 打包分发 | 支持 Windows + macOS 安装包 | P2 |
| 安装包体积 | < 150MB | P2 |

### 打包策略

**后端打包**：
- 使用 esbuild 打包成单一 `index.cjs` 文件
- 外置原生模块 `better-sqlite3` 和 `@node-rs/bcrypt`（通过 `--external` 参数）
- **不手动复制原生模块**，由 electron-builder 自动处理

**前端打包**：
- 正常 Vite 构建
- 必须使用 HashRouter（已在 Phase 2 确认）

**原生模块处理（遵循 Electron 最佳实践）**：
- 原生模块作为 `dependencies` 声明在 `packages/electron/package.json`
- electron-builder 自动识别依赖并处理：
  - Universal 架构的 lipo 合并
  - 二进制文件重编译
  - 放入 `app.asar.unpacked` 或标准 `node_modules`
- 通过 `asarUnpack` 配置防止 ASAR 读取失败

**运行时路径处理（Critical）**：
- 打包后 `index.cjs` 位于 `Resources/backend/index.cjs`
- 原生模块位于 `Resources/app.asar.unpacked/node_modules/...` 或 `app.asar` 内
- Node.js 默认路径搜索可能找不到原生模块，需要在 Electron 主进程中正确配置：
  - 方案 A：通过 `ELECTRON_RUN_AS_NODE` 环境变量启动后端子进程
  - 方案 B：设置正确的 `NODE_PATH` 环境变量
  - 方案 C：在主进程中手动 require 路径映射

## 任务列表

### 5.1 后端打包配置

**文件**: `packages/backend/package.json` (scripts 部分)

```json
{
  "scripts": {
    "build": "esbuild src/index.ts --bundle --platform=node --outfile=dist/index.cjs --external:better-sqlite3 --external:@node-rs/bcrypt"
  }
}
```

**关键点**:
- 使用 esbuild 打包成单一 `index.cjs` 文件
- 外置原生模块 `better-sqlite3` 和 `@node-rs/bcrypt`
- **移除手动复制命令**：原生模块由 electron-builder 自动处理，不需要 `build:electron` 或 `cp` 命令

**better-sqlite3 数据库路径处理**:
- `better-sqlite3` 编译后生成 `better_sqlite3.node` 文件
- 已通过 `asarUnpack: "**/*.node"` 确保正确解包
- **数据持久化路径**：生产环境下数据库路径必须使用 `app.getPath('userData')`，而非相对路径
- 示例代码：
  ```typescript
  // backend/src/database.ts
  import { app } from 'electron'
  import Database from 'better-sqlite3'
  import path from 'path'

  const getDatabasePath = () => {
    // 生产环境使用 userData 目录
    if (process.env.NODE_ENV === 'production') {
      return path.join(process.env.USER_DATA_PATH || '.', 'nanomail.db')
    }
    // 开发环境使用项目目录
    return path.join(__dirname, '../../nanomail.db')
  }

  const db = new Database(getDatabasePath())
  ```
- 主进程启动后端时需传递 `USER_DATA_PATH` 环境变量：
  ```typescript
  // electron/main.ts
  const userDataPath = app.getPath('userData')
  spawn(process.execPath, [backendPath], {
    env: { ...process.env, USER_DATA_PATH: userDataPath }
  })
  ```

**验收标准**：
- [ ] 后端可成功打包为单一文件
- [ ] 原生模块正确外置
- [ ] 打包后文件大小合理（< 5MB）
- [ ] 数据库文件在生产环境持久化到正确位置

### 5.2 electron-builder 配置

**文件**: `packages/electron/electron-builder.yml`

```yaml
appId: com.nanomail.app
productName: NanoMail
directories:
  output: release
  buildResources: build

# 只保留后端 esbuild 打包出来的代码
extraResources:
  - from: "../backend/dist/index.cjs"
    to: "backend/index.cjs"

# 让原生模块被 builder 自动处理并解包，防止 ASAR 读取失败
asarUnpack:
  - "**/*.node"

# macOS 配置
mac:
  category: public.app-category.productivity
  icon: build/icon.icns
  target:
    - dmg
    - zip
  hardenedRuntime: true
  gatekeeperAssess: false

dmg:
  contents:
    - x: 130
      y: 220
    - x: 410
      y: 220
      type: link
      path: /Applications

# Windows 配置
win:
  icon: build/icon.ico
  target:
    - nsis
    - portable

nsis:
  oneClick: false
  allowToChangeInstallationDirectory: true
  createDesktopShortcut: true
  createStartMenuShortcut: true

# 通用配置
asar: true
compression: maximum
```

**关键点**:
- 只打包 esbuild 编译后的 `index.cjs`
- **原生模块声明在 `packages/electron/package.json` 的 `dependencies` 中**
- electron-builder 自动识别并处理原生模块（Universal lipo 合并、二进制重编译）
- `asarUnpack: "**/*.node"` 确保 `.node` 文件被解包，防止 ASAR 读取失败
- macOS 启用 hardenedRuntime
- Windows 使用 NSIS 安装器

**验收标准**：
- [ ] macOS DMG 可正常安装
- [ ] Windows NSIS 安装包可正常安装
- [ ] 后端资源正确打包
- [ ] 安装包体积合理（< 150MB）

### 5.3 构建脚本

**文件**: `packages/electron/package.json`

```json
{
  "scripts": {
    "postinstall": "electron-builder install-app-deps",
    "dev": "concurrently \"vite\" \"vite build -c vite.main.config.ts --watch\" \"wait-on tcp:5173 && electron .\"",
    "build:all": "pnpm --filter @nanomail/shared build && pnpm --filter @nanomail/backend build && pnpm --filter @nanomail/frontend build && pnpm build",
    "package:mac": "pnpm build:all && electron-builder --mac --universal",
    "package:win": "pnpm build:all && electron-builder --win --x64"
  },
  "devDependencies": {
    "concurrently": "^8.0.0",
    "wait-on": "^7.0.0"
  },
  "dependencies": {
    "better-sqlite3": "^11.0.0",
    "@node-rs/bcrypt": "^1.10.0"
  }
}
```

**postinstall 脚本说明**:
- `electron-builder install-app-deps` 确保 `pnpm install` 后原生模块立即重编译
- 重编译后的原生模块与当前 Electron 版本的 ABI 匹配
- 避免 `pnpm dev` 时出现原生模块加载错误

**构建顺序**:
1. 构建 shared 模块（类型定义）
2. 构建后端（esbuild 打包，external 原生模块）
3. 构建前端（Vite 构建）
4. 构建 Electron 主进程
5. 运行 electron-builder 打包（自动处理原生模块）

**关键变更**:
- 原生模块 `better-sqlite3` 和 `@node-rs/bcrypt` 声明在 `dependencies` 中
- electron-builder 自动识别依赖，处理 Universal 架构和二进制重编译
- 移除手动复制原生模块的不可靠命令
- 使用 `concurrently` 和 `wait-on` 改进开发体验

**验收标准**：
- [ ] 构建脚本可正常执行
- [ ] 各平台打包成功

### 5.4 图标资源

**源文件**: `packages/electron/build/icon.svg`（基于 CoffeeIcon 组件）

**需要生成的图标文件**:

| 文件 | 尺寸 | 用途 |
|------|------|------|
| `build/icon.icns` | 512x512 | macOS 应用图标 |
| `build/icon.ico` | 256x256 | Windows 应用图标 |
| `build/icon.png` | 512x512 | Linux 应用图标 / 转换源 |

**图标设计**:
- 基于 `packages/frontend/src/components/icons/CoffeeIcon.tsx` 组件
- 主色调: `#2563eb` (blue-600，品牌色)
- 白色背景，圆角矩形

**生成方法**:

**方式一：使用 electron-icon-builder（推荐）**
```bash
# 安装工具
pnpm add -D electron-icon-builder

# 从 PNG 生成所有格式
npx electron-icon-builder --input=build/icon.png --output=build
```

**方式二：使用在线工具**
1. 将 `build/icon.svg` 转换为 512x512 PNG
   - 工具: https://cloudconvert.com/svg-to-png
2. 使用 https://cloudconvert.com/png-to-ico 生成 .ico
3. 使用 https://cloudconvert.com/png-to-icns 生成 .icns

**方式三：macOS 命令行**
```bash
# 创建 iconset 目录
mkdir -p build/icon.iconset

# 生成各尺寸 PNG
sips -z 16 16     build/icon.png --out build/icon.iconset/icon_16x16.png
sips -z 32 32     build/icon.png --out build/icon.iconset/icon_16x16@2x.png
sips -z 32 32     build/icon.png --out build/icon.iconset/icon_32x32.png
sips -z 64 64     build/icon.png --out build/icon.iconset/icon_32x32@2x.png
sips -z 128 128   build/icon.png --out build/icon.iconset/icon_128x128.png
sips -z 256 256   build/icon.png --out build/icon.iconset/icon_128x128@2x.png
sips -z 256 256   build/icon.png --out build/icon.iconset/icon_256x256.png
sips -z 512 512   build/icon.png --out build/icon.iconset/icon_256x256@2x.png
sips -z 512 512   build/icon.png --out build/icon.iconset/icon_512x512.png
sips -z 1024 1024 build/icon.png --out build/icon.iconset/icon_512x512@2x.png

# 生成 icns
iconutil -c icns build/icon.iconset -o build/icon.icns
```
